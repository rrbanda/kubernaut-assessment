Kubernaut: Autonomous Kubernetes Remediation

An open-source AIOps platform that closes the loop from Kubernetes alert to verified fix. An AI agent investigates live cluster state, selects a remediation from a curated workflow catalog, executes it via Tekton/Jobs/Ansible, and verifies the outcome -- all governed by OPA policies and approval gates.

Apache 2.0 Go 1.25 v1.5 OpenShift 4.18+ By Jordi Gil (Red Hat)

What happens when something breaks in your cluster

Today, when a pod crashes or a service degrades, Prometheus detects it and pages your team within a minute. That part works. But then an SRE has to manually investigate: check pod status, read logs, query metrics, trace dependencies, figure out the root cause, find the right fix, apply it, and verify it actually worked. That takes 30 to 60+ minutes per incident -- and the quality depends on who is on call and what they happen to know.

As your clusters grow, this doesn't scale. More microservices means more alerts. Same team. Alert fatigue sets in, incidents get missed, and mean time to recovery keeps climbing.

Kubernaut automates the entire process after detection: root cause analysis, fix selection, execution, and verification. But it does this differently from a fully autonomous AI agent. The AI has full freedom to investigate your cluster using read-only tools -- it can check any resource, query any metric, read any log. But when it comes to fixing things, it can only select from a catalog of workflows your team has pre-approved. The actual fix runs through Tekton, a K8s Job, or Ansible -- not through the AI. OPA policies gate every action. The AI is smart enough to figure out what's wrong, but it's never given the keys to break anything.

How Kubernaut is built

Agentic where it's safe, constrained where it's risky

Kubernaut is not a fully autonomous AI agent that generates and applies arbitrary patches to your cluster. It is a hybrid: the AI has full autonomy to investigate (37 read-only tools, open-ended reasoning, up to 40 turns), but is constrained for remediation (must select from a pre-approved workflow catalog -- it cannot generate kubectl commands or patches). Execution is deterministic (pre-tested Tekton Pipelines, K8s Jobs, or Ansible playbooks with dedicated per-workflow ServiceAccounts). Governance is policy-driven (OPA Rego evaluates every remediation, not the AI's judgment). This separation of AI intelligence from deterministic execution is what makes it enterprise-adoptable.

Kubernaut is a microservices platform written in Go. It consists of 11 independently deployed, stateless services. The platform uses two complementary communication patterns: CRD controllers watch Kubernetes custom resources and drive the remediation pipeline through status transitions (all pipeline state lives in etcd), while several services also expose REST APIs via OpenAPI specs for synchronous operations. The Gateway receives AlertManager webhooks and creates CRDs. DataStorage manages PostgreSQL persistence and exposes a REST API for audit, workflow catalog, and remediation history queries. KubernautAgent exposes REST and MCP endpoints for two agent functions: root cause investigation and workflow selection from the catalog (two separate sessions, where workflow selection consumes the output of the investigator). APIFrontend exposes MCP and A2A endpoints for external agent integration. DataStorage uses a Valkey instance as a dead-letter queue for audit events when PostgreSQL is unavailable, and all services implement graceful shutdown with audit buffer flushing to minimize audit event loss. A central controller called the RemediationOrchestrator reads those statuses and advances the pipeline by creating the next CRD in the chain.

Pipeline state lives in etcd via CRDs -- every decision the platform makes is observable via kubectl get remediationrequests. Persistent data (audit trails, workflow catalog, remediation history, effectiveness scores) lives in PostgreSQL via the DataStorage service. If any service crashes, CRD controllers restart from the last persisted state via controller-runtime reconciliation. Audit events are buffered in-memory and flushed on graceful shutdown; if PostgreSQL is unavailable, DataStorage uses Valkey as a dead-letter queue to prevent audit loss.

Observability and Investigation

The read-only part of the platform. Nothing is changed, nothing is executed. The goal is to produce a root cause analysis with a confidence score.

Signal intake and classification

When Prometheus AlertManager fires an alert, it sends a webhook POST to the Gateway service. The Gateway resolves the alert's target resource to its owner (e.g., Pod to Deployment via ownerReferences), computes a SHA256 fingerprint of namespace:kind:name, and checks whether an active RemediationRequest already exists for that fingerprint. Duplicates are counted but not re-processed. New signals become RemediationRequest CRDs.

The SignalProcessing controller then enriches the signal with namespace-level context and evaluates four OPA Rego prepared queries to classify the signal's environment, severity, priority, and custom labels. Infrastructure label detection (is this GitOps-managed? Does it have an HPA? Is there a PDB?) happens later, after the AI agent determines the RCA target resource -- those labels influence workflow selection, not signal classification.

AI-powered root cause analysis

The KubernautAgent receives the enriched signal and runs a structured investigation using an LLM (any of 9 supported providers) with access to 37 read-only tools implemented as typed Go functions via client-go. These tools include: resource get/list/describe for any API group, pod log retrieval, Prometheus instant and range queries, metrics API queries, K8s event lookups, and resource context lookups (owner chain + remediation history).

The agent calls these tools in a loop -- up to 40 turns -- following leads the way an SRE would: check pod status, read logs, look at events, query Prometheus, trace ownership, check related resources. When it has enough evidence, it submits a structured JSON result containing a root cause summary, causal chain, confidence score, and an 8-dimension due diligence assessment.

Every tool is read-only. The agent cannot create, patch, delete, or exec into any Kubernetes resource. It can only observe and analyze. This is an architectural constraint enforced in code, not a policy setting.

Remediation and Verification

The action part of the platform -- completely separated from investigation. The AI agent that investigated the root cause never directly touches the cluster. Instead, it recommends a workflow from a curated catalog, and a separate controller executes it with its own credentials.

Workflow selection from a curated catalog

After the RCA is complete, the KubernautAgent runs a second agent session (separate from the first). In this session, the agent receives the RCA findings plus enrichment data (owner chain, remediation history from previous incidents, detected infrastructure labels) and searches the workflow catalog stored in DataStorage. It proposes a workflow ID, which is validated against the catalog's JSON schema. If the proposal is invalid, the system gives the agent correction context and retries up to 3 times. If no valid workflow is found, the incident is escalated to a human with the full RCA.

The agent does not generate arbitrary patches or kubectl commands. It selects from a pre-approved catalog of workflows that your team has authored, tested, and declared as RemediationWorkflow CRDs. The AI provides intelligence; the catalog provides control.

Policy-gated approval

Before any workflow runs, a separate OPA Rego policy (approval.rego) evaluates whether the proposed remediation requires human approval. The policy can gate on confidence threshold, environment, severity, or specific workflow types. If approval is required, the system creates a RemediationApprovalRequest CRD and pauses until a human approves via Slack, the Console UI, or a direct CRD status update.

Execution through dedicated controllers

Once approved, the RemediationOrchestrator creates a WorkflowExecution CRD. The WorkflowExecution controller runs the fix through one of three execution engines: a Tekton PipelineRun (resolved from an OCI bundle), a Kubernetes batch/v1 Job, or an Ansible job template dispatched to AWX/AAP via REST API. Each workflow runs with its own dedicated ServiceAccount and short-lived TokenRequest credentials.

Effectiveness verification

After execution, the EffectivenessMonitor checks four dimensions: pod health, alert resolution in AlertManager, Prometheus metric stability, and spec hash drift. It produces a health score from 0 to 100% and persists the result. If the fix failed, the outcome is recorded in remediation history. Next time the same alert fires, the agent will see "previous workflow scored 0 -- regression detected" and avoid repeating the failed approach.

Agentic Integration (v1.5 -- available now)

What are MCP and A2A?

MCP (Model Context Protocol) is an open standard that lets AI assistants call tools on remote servers -- like a REST API, but designed for AI agents. OpenShift Lightspeed already uses MCP to query cluster resources. A2A (Agent-to-Agent) is how one AI agent delegates an entire task to another AI agent. Think of MCP as "calling a function" and A2A as "assigning a ticket."

Kubernaut is not just an autonomous pipeline -- it is also a platform that other AI tools can call. Any MCP-compatible assistant (Lightspeed, Cursor, Claude) can invoke Kubernaut to investigate and fix issues. Any A2A orchestrator can delegate full remediation tasks to it. This is implemented and functional in v1.5.

Interactive collaboration MCP

An SRE using Lightspeed or any AI chat calls Kubernaut tools directly:

Trust Ladder: observation to full autonomy

No team flips a switch to full automation on day one. Kubernaut has a progressive trust model that lets you start by watching, build confidence through data, and gradually increase autonomy as the platform proves itself.

L1v1.4
Observe

Dry-run mode. See what Kubernaut would do without any execution. Full RCA notifications with context. Zero risk.

L2v1.3+
Approve

Every remediation goes through an approval gate. OPA policy per context. Confidence scores and RCA visible before approval.

L3v1.3+
Selective Autonomy

Trusted workflows auto-approve. New ones still require human review. Effectiveness scores drive graduation.

L4v1.3+
Full Autonomy

All workflows execute without intervention. 4-dimension verification on every run. Rego policy rollback to L3 at any time.

The closed loop

Observe, investigate, select, approve, execute, verify, learn, repeat. Every cycle produces data that makes the next cycle better. Failed remediations are flagged in history. Effective workflows build trust. The system adapts without retraining the model.

Summary Verdict

Kubernaut is the most comprehensive open-source attempt at closing the loop from Kubernetes alert to verified remediation. It separates observability (AI investigation with read-only tools) from remediation (catalog-driven workflow execution with OPA governance) and connects them through a CRD-native state machine that is fully observable, crash-recoverable, and auditable.

The architecture is sound, the code quality is high, and the v1.5 agentic direction (MCP/A2A) positions it for the emerging agent-native operations paradigm.

How is this different from...?

"What about OpenShift Lightspeed?" OCP native

Lightspeed is a cluster assistant. Kubernaut is an autonomous remediation platform with conversational capabilities.

With v1.5, Kubernaut is also conversational -- it supports natural language signal intake and interactive MCP sessions. But its core differentiator is what happens after the conversation: it executes remediation workflows from a curated catalog, runs OPA approval gates, verifies fixes, and learns from outcomes. They are complementary: Kubernaut's MCP tools could be registered as a Lightspeed MCP server.

"What about the OpenShift MCP ecosystem?" OCP native

Those are infrastructure for hosting MCP servers. Kubernaut is a purpose-built MCP server that would run on that infrastructure.

Kubernaut would be deployed via the MCP Lifecycle Operator, registered in the MCP Catalog, and routed through the MCP Gateway. The OCP MCP Server gives agents read access to cluster resources. Kubernaut gives agents the ability to investigate, select a fix, execute it, and verify the outcome.

"What about agent-per-issue tools (Pulse Agent, ADK OpenShift Agent, KubeHeal)?" agents

Those let the LLM apply patches directly. Kubernaut constrains the agent to a curated catalog.

"What about Ansible Automation Platform?" Red Hat

Kubernaut already uses AAP/AWX as one of three execution engines.

AAP is the "trusted execution layer." Kubernaut is the "intelligence layer" that feeds AAP the right playbook at the right time. AAP needs something upstream to tell it what to execute and when. That is Event-Driven Ansible for rules, or Kubernaut for AI-driven investigation.

"Why not Event-Driven Ansible + Dynatrace?" architecture

That is the rule-based approach. Kubernaut handles the long tail.

Rules work for known scenarios. But when CrashLoopBackOff has 10 different root causes, a static rule cannot distinguish them. Real example: a CrashLoopBackOff caused by a broken ConfigMap pushed via ArgoCD. The agent traced Pod to logs to ConfigMap to ArgoCD annotations, determined it was GitOps-managed, and selected git-revert instead of kubectl patch.

"Is it safe to let an LLM near production?" security

The LLM never touches production. It only reads and recommends.

Every agent tool is read-only. Mutations happen through a separate WorkflowExecution controller with dedicated per-workflow ServiceAccounts.

Six defense layers (all code-verified):

1. Sanitization: credential scrubbing + prompt injection detection

2. Anomaly detection: circuit breaker on excessive tool calls

3. OPA Rego: policy gates on every remediation

4. Shadow agent: second LLM reviews each investigation step

5. Approval CRD: human gate for high-risk actions

6. Dry-run mode: full RCA without any execution
