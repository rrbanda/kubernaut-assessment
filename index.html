Minute 2 -- How It Works

The remediation pipeline: 5 phases, all CRD-native

Every phase is a Kubernetes CRD with status transitions. All state in etcd. All communication via CRD status. Standard kubectl tooling. 9 purpose-built CRDs define the entire lifecycle.

How the AI investigation works (for non-AI folks)

Imagine an SRE who can only read your cluster, never write. They run kubectl get, check logs, query Prometheus, follow owner references -- in a loop -- until they find the root cause. Then they look through your approved runbook catalog and recommend the right fix. That is what the AI agent does, except it's software calling 36 typed Go functions against the K8s API. It cannot create, patch, or delete anything. It only reads and recommends.

Minute 3 -- Building Trust

Trust Ladder: observation to full autonomy

Teams don't flip a switch to full automation. Kubernaut has a progressive trust model -- start by watching, graduate to full autonomy when the data proves the platform is reliable.

L1v1.4
Observe

Dry-run mode. See what Kubernaut would do without any execution. Full RCA notifications with context.

L2v1.3+
Approve

Every remediation goes through an approval gate (RAR). OPA policy per context. Confidence scores visible.

L3v1.3+
Selective Autonomy

Trusted workflows auto-approve. New ones require human review. Effectiveness scores drive graduation.

L4v1.3+
Full Autonomy

All workflows execute without intervention. 4-dimension verification. Rego policy rollback to L3 at any time.

The LLM learns from failure

When a fix doesn't work (healthScore = 0), the effectiveness feedback is injected into future investigation context. The LLM sees "previous workflow scored 0 -- regression detected" and avoids repeating failed approaches. When it can't find a good fix, it escalates to a human.

Minute 5 -- What's Next

v1.5 Agentic Integration (Q2 2026)

Quick primer: what are MCP and A2A?

MCP (Model Context Protocol) is an open standard that lets AI assistants call tools on remote servers -- like a REST API, but designed for AI agents. OpenShift Lightspeed already uses MCP to query cluster resources. A2A (Agent-to-Agent) is how one AI agent delegates an entire task to another AI agent and gets results back. Think of MCP as "calling a function" and A2A as "assigning a ticket."

Kubernaut is evolving from an autonomous pipeline to a platform that other AI tools can call. Any MCP-compatible assistant (Lightspeed, Cursor, Claude, etc.) can invoke Kubernaut to investigate and fix issues.

Interactive: "Let's investigate together" MCP

An SRE using Lightspeed or any AI chat can call Kubernaut tools directly:

investigate -- Start an interactive root cause analysis with live streaming

select_workflow -- Browse the workflow catalog and pick a fix

complete_no_action -- Close the case if no remediation is needed

The human stays in control. The AI does the digging.

Autonomous: "Fix this, tell me when done" A2A

A monitoring agent or orchestrator delegates the entire job to Kubernaut:

1. Sends alert or natural language description

2. Kubernaut investigates, selects fix, executes, verifies

3. Returns verified result or escalates with full RCA report

No human in the loop unless the policy requires it.

What to tell your audience Verdict

Kubernaut is the most comprehensive open-source attempt at closing the alert-to-remediation loop with AI. It's not a diagnostic tool that stops at "here's what's wrong" -- it investigates, selects a fix, executes it, and proves it worked. That full depth is unique in the OSS space.

The engineering is genuinely impressive for a solo-authored project: proper CRD patterns, extensive testing, real safety mechanisms. The trust ladder (observe/approve/selective/full autonomy) is the right adoption model for enterprise teams.

The risk is clear: one maintainer, early stage, and some gaps between documentation claims and code reality. But the architecture is sound, the code quality is high, and the v1.5 agentic direction (MCP/A2A) positions it well for the emerging agent-native operations paradigm.

Anticipated Questions

How is this different from...?

Crisp answers for the questions your audience will ask.

"What about OpenShift Lightspeed?" OCP native

Lightspeed is a conversational assistant. Kubernaut is an autonomous remediation pipeline.

Lightspeed answers questions about your cluster ("what pods are failing?") and can query resources via its Observability MCP server. But it does NOT: execute remediation workflows, maintain a workflow catalog, run OPA approval gates, verify fixes worked, or learn from outcomes. It's a Q&A chatbot with cluster read access. Kubernaut is a closed-loop remediation engine.

Complementary, not competitive: Kubernaut's MCP tools could be registered AS a Lightspeed MCP server. An SRE could ask Lightspeed "investigate this alert" and Lightspeed would delegate to Kubernaut via MCP. The v1.5 RHDH Console Plugin goes further -- embedding Kubernaut directly in Developer Hub.

"What about the OpenShift MCP Server, MCP Gateway, MCP Lifecycle Operator, MCP Catalog?" OCP native

Those are infrastructure for MCP servers. Kubernaut is a consumer AND provider of MCP.

The OpenShift MCP ecosystem (MCP Server for OCP, MCP Gateway via Connectivity Link, MCP Lifecycle Operator, MCP Catalog in OpenShift AI 3.4) provides the platform for deploying and managing MCP servers. Kubernaut would run ON this infrastructure -- its MCP server deployed via the MCP Lifecycle Operator, registered in the MCP Catalog, routed through the MCP Gateway. Kubernaut is a purpose-built remediation MCP server; the OCP MCP stack is the hosting layer.

The OCP MCP Server gives agents read access to cluster resources. Kubernaut gives agents the ability to investigate, select a fix, execute it, and verify the outcome -- with governance at every step.

"What about agent-per-issue patterns? Pulse Agent, ADK OpenShift Agent, KubeHeal?" agents

Those are single-agent tools. Kubernaut is a platform with a structured pipeline.

Key difference: agent-per-issue tools let the LLM generate and apply arbitrary patches. Kubernaut constrains the LLM to selecting from a curated workflow catalog -- the LLM investigates and recommends, but execution is through pre-tested Tekton/Job/Ansible workflows. This is the "AI intelligence + deterministic execution" pattern Red Hat's own AAP team advocates.

"What about Ansible Automation Platform as the trusted execution layer?" Red Hat

Kubernaut already uses AAP/AWX as one of three execution engines.

Red Hat positions AAP as the "trusted execution layer for AIOps" -- AI recommends, Ansible executes. Kubernaut implements exactly this pattern: the LLM recommends a workflow, OPA gates approval, and execution runs via Tekton OR K8s Jobs OR Ansible/AWX. The Ansible executor in Kubernaut calls AWX job templates via REST API with credentials and polling.

The difference: AAP needs something upstream to tell it WHAT to execute and WHEN. That's Event-Driven Ansible for rule-based triggers, or Kubernaut for AI-driven investigation + selection. Kubernaut is the "intelligence layer" that feeds AAP the right playbook at the right time.

"Why not just use Event-Driven Ansible + Dynatrace/ServiceNow?" architecture

That's the rule-based approach. Kubernaut handles the long tail.

EDA + Dynatrace/ServiceNow is the "thermostat" pattern: if CPU exceeds 90%, run scale-up playbook. Works great for known, deterministic scenarios. But when the same alert (CrashLoopBackOff) has 10 different root causes -- bad config, OOM, dependency failure, cert expiry, image pull error -- a static rule can't distinguish them. Kubernaut's LLM investigates the actual cause via 36 tools, then selects the right workflow from the catalog.

Real example from the Kubernaut demo: a CrashLoopBackOff where the root cause was a broken ConfigMap pushed via ArgoCD. The LLM traced Pod to logs to ConfigMap to ArgoCD annotations, determined it was GitOps-managed, and selected git-revert (not kubectl patch). A rule-based system would need a specific rule for this exact scenario.

"Is it safe to let an LLM touch production?" security

The LLM never touches production. It only reads and recommends.

Every single agent tool is read-only (verified from code). The LLM cannot create, patch, delete, or exec into anything. It investigates via read-only K8s API calls and proposes a workflow ID from the catalog. The actual mutation happens through a separate WorkflowExecution controller that runs pre-approved Tekton Pipelines or K8s Jobs with dedicated per-workflow ServiceAccounts.

Defense layers (all code-verified):

1. Sanitization pipeline: credential scrubbing, injection detection

2. Anomaly detection: circuit breaker on tool abuse

3. OPA Rego: policy gates on approval

4. Shadow agent: second LLM reviews each step

5. Approval CRD: human gate for high-risk actions

6. Dry-run mode: see what it would do without execution
